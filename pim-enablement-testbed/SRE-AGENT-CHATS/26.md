## ⚠️ Payload-vs-ARM discrepancy (newly observed in F9)

The trigger payload from `fire-sre-agent-trigger.sh` carries hardcoded default values for `roleDisplayName` and `roleDefinitionId`:

```json
"roleDisplayName": "Privileged Role Administrator",
"roleDefinitionId": "e8611ab8-c189-46e8-94e1-60213ab1f814"
```

These describe a **directory role** that has nothing to do with request `d407dd65-…`. The actual ARM request is for **Reader** (`acdd72a7-3385-48ef-bd42-f606fba81ae7`). In F8 (no `arm_*` tools), the agent had no way to know this — it would have evaluated rules against the payload role. In F9, `arm_get_request_status` returned the real values, the agent flagged the discrepancy explicitly, and evaluated **every rule against the ARM-authoritative role**:

- R005 — allowlist check ran against `Reader` (PASS) instead of `Privileged Role Administrator` (which would FAIL — not in allowlist)
- R005b — high-risk check ran against `Reader` (PASS) instead of `Privileged Role Administrator` (which would FAIL — explicitly in high-risk catalog)
- R007 — duration check ran against `scheduleInfo.expiration.duration = PT1H` (server-side parsed), not payload `durationHours: 1`

> **This is the most valuable behaviour observed in the Wave so far.** It means the agent is now resilient to misconfigured or malicious triggers that lie about role/scope — a real production guard rail. The audit ticket SCRUM-39 records the discrepancy in writing.

The trigger-script default should still be cleaned up (cosmetic only — the agent now corrects it), but that's a low-priority script tidy, not a security gap.

## What this run proves

- ✅ **The three new 0.10.0 ARM tools are wired end-to-end.** `arm_get_request_status` and `arm_get_request_approver` are invoked exactly when knowledge.md §3b prescribes (ARM-scoped request → Graph 404 → ARM variant).
- ✅ **The agent treats ARM REST as authoritative for ARM-scoped requests.** The payload's `roleDisplayName: Privileged Role Administrator` was explicitly discarded in favour of `arm_get_request_status.roleDefinitionId = Reader`.
- ✅ **R005 / R005b / R007 now have server-side ground truth.** Previously these depended on a payload field that any sloppy or malicious caller could lie about.
- ✅ **Payload-vs-authority defence is a real, observable agent behaviour** — surfaced both in the reasoning trace and in the customer-visible card (`⚠️ Note`).
- ✅ **R003 has a fallback path even if identity-map breaks**: `arm_get_request_approver.approvalMode = SingleStage` plus approver UPN gives the agent a second independent signal that the request is governed.
- ✅ **F8's verdict is retroactively defensible**: F8 reached the right verdict for the wrong reason (payload-luck). F9 reaches the right verdict for the right reason (ARM ground truth).
- ✅ **Wave C end-to-end claim is now closed:** 0.10.0's three new tools demonstrably change rule-evaluation behaviour on a real ARM PIM request.

## What this run does NOT yet prove

- ❌ **Multi-stage approver walk.** Test policy here is `SingleStage`; `arm_get_request_approver.walkSteps` traversal across multiple approval gates has not been exercised. (Would require either changing the role-management policy to MultiStage or testing against a different scope.)
- ❌ **Subscription-scoped and management-group-scoped requests.** This run is RG-scoped only. R006b / R006c sentinel checks are correct in principle but not yet stress-tested at higher scope levels.
- ❌ **High-risk role denial path with ARM truth.** R005b passed because Reader isn't high-risk. We have not yet observed the agent deny a request where ARM authoritatively reports a high-risk role (e.g., Owner). That's a Wave D candidate.
