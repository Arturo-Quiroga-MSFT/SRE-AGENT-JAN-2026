> **🆕 R007 CANNOT VERIFY — real architectural gap, not a regression.** The PIM request lives on **ARM** (`/subscriptions/.../providers/Microsoft.Authorization/roleAssignmentScheduleRequests/...`), but `pim-mcp` only queries **Microsoft Graph** (`/roleManagement/directory/roleAssignmentScheduleRequests/...`). All three Graph lookups (`get_role_definition`, `get_request_status`, `get_request_approver`) returned 404 because directory-scope and resource-scope PIM are separate API surfaces. Without `get_request_status`, the agent can't read the request's `scheduleInfo.expiration.duration` to evaluate R007. Push-mode payload doesn't include duration either. **pim-mcp needs an ARM-scoped variant of these tools** for Azure-resource PIM evaluations to be complete.

## What this run proves

- ✅ **End-to-end real PIM submission works.** First time we've put a non-synthetic, real ARM PIM request through the agent. Setup script, activate script, and trigger script all chained cleanly.
- ✅ **Identity-map R003 PASS holds across schemes.** Same UPN → Atlassian accountId mapping that worked in 20.md/21.md works here for a brand-new ticket.
- ✅ **Agent gracefully degrades on Graph 404s.** Three pim-mcp tools 404'd (role lookup, status lookup, approver lookup) because the request is ARM-scoped — agent recognized this, fell back to payload data for what it could, and explicitly reported `CANNOT VERIFY` for R007 instead of hallucinating PASS.

## What this run does NOT yet prove

- ❌ **An Approve verdict is reachable** — three blockers still in the way:
  1. The R001d allowlist needs to include `Task` (or another testbed → canonical mapping). Local YAML edit done; **needs commit + push**.
  2. The R006 allowlist needs the real subscription prefix, not `PLACEHOLDER-sub-id`. Local YAML edit done; **needs commit + push**.
  3. R007 needs an ARM-scoped duration source (either pim-mcp ARM tool, or include duration in trigger payload, or have the activate script POST it back to the agent post-submit).
