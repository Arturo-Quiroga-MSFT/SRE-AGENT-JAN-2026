## Root cause of `arm_*` skip + next-step F9

**Root cause:** `knowledge.md §3b` is worded advisorily — the agent treats it as a *fallback explanation* ("404 was expected per §3b") rather than as a *routing precondition* ("inspect scope first; MUST call `arm_*` for `/subscriptions/` scopes").

**F9 fix plan:**

1. **Tighten knowledge.md §3b** to imperative MUST/MUST NOT language. Concrete change:

   > **§3b (revised):** Before calling any of `get_role_definition`, `get_request_status`, or `get_request_approver`, you **MUST** inspect `directoryScopeId` (alias `scope`):
   > - If it starts with `/subscriptions/` or `/providers/Microsoft.Management/managementGroups/`, you **MUST** call the `arm_*` variant (`arm_get_role_definition`, `arm_get_request_status`, `arm_get_request_approver`) and you **MUST NOT** call the Graph-endpoint variant for that lookup.
   > - Otherwise (scope is `/`, `/administrativeUnits/<id>`, or `/roleManagement/directory`), you **MUST** call the Graph-endpoint variant and you **MUST NOT** call the `arm_*` variant.
   > - The previous "Graph-first, then fall back to payload" pattern is **deprecated** for ARM-scoped requests.

2. **Re-upload `knowledge.md`** as a standalone Knowledge file (same precedence trick that fixed F6 → F7).

3. **Fire F9** as an exact re-fire of F8 (same `REQUEST_ID=d407dd65-…` if still `PendingApproval`, otherwise re-submit). Expected:
   - Same 14/14 PASS verdict
   - Tool-call table now shows `arm_get_role_definition` / `arm_get_request_status` / `arm_get_request_approver` instead of (or in addition to) the Graph 404s
   - R007 sourced from `arm_get_request_status.scheduleInfo.expiration` (server-side parsed) rather than payload `durationHours`
