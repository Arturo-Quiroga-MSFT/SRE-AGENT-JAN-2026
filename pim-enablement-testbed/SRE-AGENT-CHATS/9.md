## Tier 1 — Directly unblocks validation rules currently emitting "REVIEW MANUALLY"

### 1. `get_user_group_memberships(principal_id)` → list of groups

**Unblocks:** Rule **R004** (group membership check — hard rule)

This is the single biggest gap called out explicitly in [knowledge.md](../agent/knowledge.md) line 193: *"User's transitive group memberships (R004) — ⚠ NOT COVERED"*. Today R004 always emits REVIEW MANUALLY, which degrades every verdict's confidence.

| Field | Value |
|---|---|
| Graph endpoint | `GET /users/{id}/transitiveMemberOf?$select=id,displayName,groupTypes` |
| Permission needed | `GroupMember.Read.All` (application) — low risk, read-only |

### 2. `list_recent_activations(principal_id, window_hours=24)` → recent activation instances

**Unblocks:** Rule **R008** (activation frequency — soft rule)

The knowledge.md notes this is only *"partial via `list_active_role_assignments`"* because that tool shows currently active assignments, not historical activations that have already expired. If a user activated 3 times today but all expired, the current tools see 0.

| Field | Value |
|---|---|
| Graph endpoint | `GET /roleManagement/directory/roleAssignmentScheduleInstances?$filter=principalId eq '{id}'&$orderby=startDateTime desc&$top=20` |
| Alternative | Filter `list_pim_request_history` by `principal_id` + `status=Provisioned` and count within the window |
| Permission needed | None (existing perms) |

## Tier 2 — Improves audit quality and compliance posture

### 3. `get_user_license_assignments(principal_id)` → assigned SKUs/service plans

**Why:** Zafin operates in a banking-compliance context. Several PIM scenarios only make sense if the requester has the right license (e.g., Entra ID P2 for PIM itself, E5 for advanced security roles). A license check would catch:

- Users requesting roles they can't effectively use
- Validation that the service-account user still has P2 (a hard prerequisite)

| Field | Value |
|---|---|
| Graph endpoint | `GET /users/{id}/licenseDetails?$select=skuPartNumber,servicePlans` |
| Permission needed | `User.Read.All` (already granted) |

### 4. `get_directory_audit_logs(principal_id?, activity?, time_range_hours=24)` → audit events

**Why:** PIM justifications often reference a recent event (*"secret leak detected", "5xx spike after deploy"*). Today the agent takes the requester's word for it. With audit logs, the agent could verify the claim — e.g., confirm that an app credential was actually modified, or that a role assignment change actually happened.

| Field | Value |
|---|---|
| Graph endpoint | `GET /auditLogs/directoryAudits?$filter=activityDateTime ge {start} and targetResources/any(t: t/id eq '{id}')&$top=25` |
| Permission needed | `AuditLog.Read.All` (application) — slightly elevated but read-only |

### 5. `get_sign_in_activity(principal_id, time_range_hours=24)` → recent sign-ins

**Why:** Detects anomalous patterns before approving — e.g., requester signing in from an unusual location or device right before requesting Tier 0 access. Also useful for the service-account monitoring mentioned in [threat-model.md](../threat-model.md) Residual Risk #3.

| Field | Value |
|---|---|
| Graph endpoint | `GET /auditLogs/signIns?$filter=userId eq '{id}' and createdDateTime ge {start}&$top=10&$select=createdDateTime,ipAddress,location,status,appDisplayName` |
| Permission needed | `AuditLog.Read.All` (same as #4 — no incremental grant) |

## Tier 3 — Operational convenience and future-proofing

### 6. `resolve_scope(directory_scope_id)` → human-readable scope

**Why:** PIM scopes like `/` or `/administrativeUnits/{id}` are opaque GUIDs. Today the agent says *"scope: /"* and the approver has to know that means "entire directory." A resolver would translate:

| Raw scope | Human-readable |
|---|---|
| `/` | "Entire directory (tenant-wide)" |
| `/administrativeUnits/{id}` | "Administrative Unit: Contoso-EMEA" |
| Resource-scoped roles | Subscription / RG / resource name |

Especially important when Zafin's real rule table (D1) arrives and scope validation becomes granular.

### 7. `get_pim_policy(role_definition_id)` → PIM policy settings

**Why:** Each role's PIM policy defines max activation duration, whether approval is required, designated approvers, whether MFA is enforced, etc. Today **R007** (8-hour duration cap) is hardcoded in YAML, but the real ceiling is the PIM policy. The agent could:

- Validate that the requested duration doesn't exceed the policy max
- Confirm that approval is actually required (some roles may be auto-approved)
- Show the designated approver list in the Adaptive Card

| Field | Value |
|---|---|
| Graph endpoint | `GET /policies/roleManagementPolicies?$filter=scopeId eq '/' and scopeType eq 'DirectoryRole'` then `GET /policies/roleManagementPolicies/{id}/rules` |
| Permission needed | `RoleManagementPolicy.Read.Directory` (application) |

### 8. `post_teams_adaptive_card(webhook_url, card_payload)` → delivery confirmation

**Why:** This is the **F3** gap from the README. Today the Adaptive Card payload is emitted inline in the SRE Agent chat and humans manually copy it to Teams. A dedicated tool would close the loop end-to-end.

| Field | Value |
|---|---|
| Endpoint | Teams Incoming Webhook URL (not Graph) |
| Permission needed | None |
