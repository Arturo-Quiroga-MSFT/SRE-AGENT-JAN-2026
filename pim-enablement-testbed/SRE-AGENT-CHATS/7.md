## What we can fetch today

| Tool | History capability |
|---|---|

| `get_request_approver(request_id)` | ✅ Full audit trail for a known request — approver identity, timestamp, justification. |
| `list_active_role_assignments(principal_id)` | ⚠️ Currently-active only — not historical. |
| `list_eligible_role_assignments(principal_id?)` | ⚠️ Current eligibilities only — not historical. |
| `list_pending_pim_requests` | ❌ Filters to `status eq 'PendingApproval'` — hides everything else. |

## The gap

The `list_pending_pim_requests` response surfaced `fetchedCount: 6` — Graph holds **6 total** `roleAssignmentScheduleRequests` for the tenant, but the tool returns only the **1** in `PendingApproval`. The other 4 request IDs are invisible to the agent until something hands them their IDs.

### Proposed new tool — `list_pim_request_history`

```python
list_pim_request_history(
    principal_id: Optional[str] = None,   # filter by requester
    status: Optional[str] = None,         # Provisioned / Denied / Cancelled / Expired
    top: int = 25,
)
```

…calling:

```http
GET /roleManagement/directory/roleAssignmentScheduleRequests
    ?$filter=status ne 'PendingApproval'
    &$orderby=createdDateTime desc
    &$top=25
```

| Concern | Status |
|---|---|
| Graph permission | ✅ Same `RoleAssignmentSchedule.ReadWrite.Directory` already granted to the MI (UPSTREAM_BUGS BUG-001) — no new app-role grant. |
| Read-only-by-construction guarantee | ✅ Preserved — new tool only LISTs. |
| Backward compatibility | ✅ Additive — existing 8 tools unaffected. |
| Bumps `pim-mcp` to | `0.8.0` |

## Bottom line

| Question | Answer |
|---|---|
| Can we fetch the disposition of a known historical request? | ✅ Yes — `get_request_status` + `get_request_approver` today. |
| Can we browse all historical requests? | ❌ Not today — needs a new tool (`list_pim_request_history`). |
| Is the data available in Graph? | ✅ Yes — `roleAssignmentScheduleRequests` returns all statuses; current tool just filters them out. |
