## 🛑 Decision: deprecate the scheduled task in favor of the HTTP trigger

This is the **last scheduled-task evaluation captured for `pim-recommender`**. Going forward, the scheduled task is being disabled and the HTTP trigger (15.md push-mode path) is the production entry point.

### Why the HTTP trigger is the better practice

| Concern | Scheduled task (pull) | HTTP trigger (push) |
|---|---|---|
| **Latency** | Up to full cron interval (5 min default; tightening multiplies cost & noise) | ~1s from event → agent thread |
| **Token cost** | Burns model tokens every tick whether or not anything is pending. ~288 ticks/day × empty enumeration | Burns tokens only on real events |
| **Re-eval noise** | Same request gets re-evaluated every tick until approved/denied. `SCRUM-23` already has comment `10140` for one request — the cron will keep appending until it resolves | One evaluation per real Graph event; no spurious re-eval |
| **Audit trail** | Multiple comments on the same ticket per request lifecycle → noisy; harder to correlate "what did the agent decide *first*" | One ticket per event; clean 1:1 mapping |
| **Operational visibility** | Failure modes hide inside scheduled-task logs; a stuck/missing enumeration goes silent | Each push has `executionTime` + `threadId` + HTTP status; trivially observable |
| **Authorization model** | Agent enumerates everything its MI can see — larger blast radius if MI is over-permissioned | Agent only sees the requestId pushed in — tighter least-privilege contract |
| **Production fit** | Anti-pattern per Deepthi's earlier critique (and standard event-driven design guidance) | Aligned with Azure SRE Agent's documented production trigger surface |
| **Concurrency** | Single-threaded loop over `list_pending_pim_requests`. N pending = N sequential evaluations in one run | One thread per event; parallel by default |

### What to do now

1. **Disable** the `pim-pending-request-evaluator` scheduled task in the SRE Agent portal (Tasks → row → toggle off). Do **not** delete — keep as a fallback that can be re-enabled.
2. **Promote the HTTP trigger** (`pim-request-created`, ID `2c2f62e3-…`) to the production entry point.
3. **Wire Wave E proper:** deploy `http-trigger-bridge/` Function with the Graph subscription so push-mode events arrive automatically (not via manual curl). 15.md proves the trigger half end-to-end; Wave E proper is the Graph→Function half.
4. **Update `WAVE-ROADMAP.md`** — Wave A row → ✅ with links to 15.md and 16.md, note that the scheduled task is deprecated.
5. **Merge `wave-a-vocab-routing` → `main`** (fast-forward).
